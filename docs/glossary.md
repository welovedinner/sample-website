# 聚餐小助手 · 核心概念与名词解释

> 本文用于统一产品 / 数据 / 实现层对关键概念的理解，  
> 避免把 UI 分组、数据库表、现实语义混为一谈。

---

## A 组 / B 组（产品层概念）

### B 组（想法池 / 聚餐对象池）

**定义：**  
B 组用于收集「未来可能发生聚餐的对象」，但**不承诺一定会发生**。

它解决的是：

- 群聊中零散出现的「这家店不错」「以后可以去」
- 低意愿、低责任、低门槛的兴趣表达

**关键特征：**

- 创建成本极低（一句话即可）
- 不需要时间
- 不隐含组织责任
- 允许信息不完整、后续补充
- 可以被多人评论、附图、补链接
- 可以被多次引用

**典型示例：**

- "年后可以去吃寿喜烧"
- "这家披萨我之前吃过不错"
- "有家无烟餐厅，适合 X"

> ⚠️ 注意：  
> B 组不是「候选聚餐」，而是**可被复用的聚餐对象 / 想法记录**。

---

### A 组（聚餐事件 / 一次具体发生的事）

**定义：**  
A 组表示「一次具体的聚餐事件」，即 **某天、某次、要不要去**。

它解决的是：

- "这次到底聚不聚"
- "现在进行到哪一步了"

**关键特征：**

- 必须有时间
- 有明确的进行状态（open / done）
- 会结束或过期
- 可以引用 B 组中的对象
- 允许地点暂时模糊

**典型示例：**

- "2 月 16 号下午包饺子"
- "元宵节当天聚一下"

> ⚠️ 注意：  
> A 组关注的是「这一次」，  
> B 组关注的是「可以反复发生的对象」。

---

## Place（聚餐对象 / 想法对象）

**定义：**  
Place 是对 B 组的抽象建模，表示一个**可被讨论、补充、引用的聚餐对象**。

它可以是：

- 某一家店
- 某种吃法（包饺子 / 寿喜烧）
- 一个模糊提议（"年后聚一下"）

**特性：**

- 本身不包含时间
- 不承诺发生
- 可以被多个 Event 引用（多对一）
- 是一个"知识对象"，而不是"行为"

---

## Event（聚餐事件）

**定义：**  
Event 表示一次具体的聚餐尝试，是 A 组的技术抽象。

**核心语义：**

> "在这个时间点，这件事要不要发生"

**特性：**

- 时间是硬约束
- 地点可以自由填写或引用 Place
- 有生命周期（创建 → 进行 → 结束）
- 是一个"状态对象"

---

## Comment（补充 / 记录）

**定义：**  
Comment 是对 Place 的补充信息，允许任何人追加内容。

**用途：**

- 地址补充
- 外部链接
- 图片记录
- 经验说明

**特性：**

- 时间序追加
- 不追求结构化
- 可带图片
- 不要求"整理完成"

---

## Participation（参与关系）

**定义：**  
Participation 表示某个人对某次 Event 的态度。

**常见态度：**

- join（参加）
- interested（感兴趣）
- maybe（可能）

**设计原则：**

- 表达意向 ≠ 承诺
- 允许修改
- 不强制唯一
- 不承担组织责任

---

## 用户（伪用户 / Pseudo User）

**定义：**  
系统不引入真实账号体系，用户通过昵称参与。

**特性：**

- 无注册、无登录
- 昵称存储在 localStorage
- 所有记录直接写入 display_name
- 信任基于现实社群，而非技术约束

---

## 一个重要的设计原则

> **本产品区分"记忆"与"行动"**

- B 组 / Place / Comment  
  → 用于保存群体的**共享记忆**

- A 组 / Event / Participation  
  → 用于描述一次**正在发生或即将发生的行动**

UI、数据结构、功能设计都应围绕这一分离展开。
